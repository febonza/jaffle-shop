dim_customers

**Grain:** one row per customer per attribute version (SCD-2).
**Used by:** `fct_orders.customer_key`, churn/LTV marts.
**AI workflow context:** to count "current customers" filter `where is_current`; to count "customers as of date X" filter `where valid_from <= X and (valid_to is null or valid_to > X)`.

customer_key
Surrogate key (`md5(customer_id || valid_from)`). Unique across the SCD-2 history.
customer_id
Stable natural key from jafgen. Not unique in this table ŌĆö repeats once per version.
customer_segment
**Represents:** computed segment in `{inactive, new, returning, vip}`.
**Business rules:** `vip` = lifetime_orders Ōēź 20 OR lifetime_revenue_usd Ōēź 500; `returning` = lifetime_orders Ōēź 5; `new` = Ōēź 1 order; `inactive` = 0 orders.
**AI workflow context:** segments transition over time as customers accumulate orders ŌĆö that's *why* this dimension is SCD-2.
is_active
True if the customer has ordered within the last 90 days *as of the build time*.

fct_order_items

**Grain:** one row per (order, product).
**Used by:** product-revenue rollups, margin analysis.

order_item_id
No description.
order_id
No description.
gross_revenue_usd
**Grain:** per line item.
**Represents:** list-price gross revenue (qty ├Ś list_price). Quantity is currently always 1 ŌĆö jafgen's `raw_order_items` does not include a quantity column; if a future generator does, update the staging model and this doc block together.
cogs_usd
**Grain:** per line item.
**Represents:** cost of goods sold, in USD. Pulled from `dim_products.unit_cost_usd`, which sums `stg_jaffle__supplies.supply_cost_usd` per SKU.
**Trade-offs:** jafgen does not emit a real BOM; the suppliesŌåÆsku map is one-to-one and represents a synthetic cost only. Margin numbers are illustrative, not real.
margin_usd
gross_revenue_usd ŌłÆ cogs_usd. Same illustrative-only caveat as `cogs_usd`.
analytics

fct_orders

**Grain:** one row per completed order.
**Source:** built incrementally from int_orders__enriched on `ordered_at`.
**Used by:** the FastAPI `/api/overview/kpis` and `/api/overview/revenue-series` endpoints; LTV/churn marts downstream.

order_id
**Grain:** per order.
**Represents:** the jafgen-issued UUID of the order.
**Used by:** every downstream join touching orders.
**Business rules:** unique, never null.
**AI workflow context:** prefer this over `(customer_id, ordered_at)` for joins ŌĆö that pair is not guaranteed unique.
ordered_at
**Grain:** per order.
**Represents:** timestamp (UTC, jafgen-naive) the order was placed.
**Business rules:** drives the incremental watermark; downstream date filters should use `ordered_date` for partitioning, `ordered_at` for ordering.
ordered_date
**Grain:** per order.
**Represents:** date-only projection of `ordered_at`, used as the partition key and the join into `dim_dates`.
**AI workflow context:** when building daily revenue, group by this ŌĆö not `ordered_at::date` ŌĆö for clarity.
date_key
**Grain:** per order.
**Represents:** integer date key in `YYYYMMDD` form (the same format used by `dim_date.date_key`).
**Used by:** star-schema joins to `dim_dates`.
customer_key
**Grain:** per order.
**Represents:** surrogate key into the current row of `dim_customers`.
**Business rules:** points to the customer's state as of *order time*; for snapshots of historic customer state, use `dim_customers.customer_key` with `valid_from` Ōēż `ordered_at` Ōēż `valid_to`.
**Excludes:** orders from customers absent from `dim_customers` are kept with a null `customer_key` (no inner join) so revenue totals don't silently drop rows. Investigate any null customer_key during QA.
store_key
Surrogate key into `dim_stores`. Used by the store-performance grid on Overview.
subtotal_usd
**Grain:** per order.
**Represents:** pre-tax order subtotal, in USD.
**Business rules:** equals sum of line-item gross revenue *minus* any order-level promotions (jafgen does not currently emit promos, so today subtotal = sum of line items).
**Excludes:** tax; tips; refunds (jafgen has none).
tax_usd
**Grain:** per order.
**Represents:** tax amount paid, in USD, computed by jafgen using the store's `tax_rate`.
order_total_usd
**Grain:** per order.
**Represents:** gross sale amount paid by the customer, in USD, *after* tax.
**Used by:** GMV KPIs, daily revenue series, LTV calc.
**Business rules:** equals `subtotal_usd + tax_usd`.
**Trade-offs:** refunds and returns are not modeled (jafgen does not produce them); this field overstates net revenue if you later layer a returns dataset. The doc block in `mart_daily_revenue` repeats this caveat.
**Excludes:** refunds; tips; cancellations.
**Includes:** all orders jafgen emits (jafgen does not emit cancellations).
**AI workflow context:** for headline revenue numbers prefer this over aggregations of `fct_order_items.gross_revenue_usd` (which sums to pre-tax).
item_count
Number of line items on the order. Joined up from `fct_order_items`.
analytics

mart_daily_revenue

**Grain:** one row per calendar day in the order window.
**Represents:** daily GMV / orders / unique customers, gap-filled against `dim_dates`.
**Used by:** Overview revenue chart, daily-trend KPI deltas.
**Trade-offs:** see fct_orders.order_total_usd ŌĆö refunds are excluded because the source lacks them.
